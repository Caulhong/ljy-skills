## 业务背景知识（用于判断异常，不用于直接输出结论）

这些是业务规律，帮助判断数据是否正常。**实际结论以数据计算值为准**，若实际值与背景知识偏差过大，应标注为数据异常并说明。

**折旧规律：**
- 折旧期约4年（48个月），未满4年时折旧额有一定波动（非精确线性）
- 超过4年后折旧费基本归零，残值极小
- 折旧不受 AZ 影响，同资源类型不同 AZ 的折旧应基本一致；差异过大（>15%）为异常

**成本结构参考：**
- 未折旧完的服务器：算力成本通常占总成本的 35~50%
- 已折旧完（≥4年）：算力成本接近0，非算力成本占主导
- 非算力中，DC 成本通常是最大分项；DCN 和网络设备成本相对较小
- 若某 AZ 的 DC 成本比同 region 其他 AZ 高出很多，通常与机房位置/电力/制冷条件有关

**AZ 成本差异参考：**
- 同 region 不同 AZ 的成本差异主要来自非算力（DC 成本）
- 一定程度的 AZ 间差异属正常；若差异极小（<3%），可能数据缺乏区分度；若极大（>50%），应核查数据准确性

**毛利与台数关系：**
- 通常台数多的资源池，边际成本摊薄，毛利应不低于台数少的同类资源池
- 台数多但毛利显著偏低（低于同类中位），说明有经营问题，需定位根因
- 根因一般为：成本偏高（对比同类 AZ）或流水偏低（对比同类分配率）

**分配率与 CPU 使用率：**
- 分配率高而 CPU 使用率低，说明客户购买了资源但未充分使用
- 这类资源池通常有超卖（Oversell）空间，但超卖比例需结合实际 CPU 水位评估
- 超卖比例过高（>2x）有服务风险，需配合监控机制

**阶段含义：**
- 起步（<1年）：处于拉新阶段，台数少，成本高
- 主力售卖（1~4年）：核心分析对象，台数多，折旧正常摊销
- 存量经营（5~7年）：折旧已完成，非算力成本为主，若流水稳定则毛利较高
- 退出整合（>7年）：折旧归零，需评估是否继续运营

## 分析维度与方法

### 分析1：同 Region 不同 AZ 成本差异 + 非算力成本拆解

**目的**：找出 AZ 间成本差距，并定位差异的成本分项根因。

**分析步骤：**
1. 选取主力售卖期数据，按 region 计算各 AZ 单台成本（cost / server_count）
2. 找 AZ 间差异最显著的几个 region（选 DC 成本差异最大的 3 个展示）
3. 拆解成本结构：分别计算折旧、DC、DCN、网络四项的单台均值
4. 对比折旧是否随 AZ 变化（理论上应接近，差异大说明数据异常）
5. 计算 az1 相对其他 AZ 的 DC 成本溢价，从数据中得出实际溢价比例
6. 基于实际数据给出结论：是否值得迁移、迁移预期收益是多少

**结论撰写原则：**
- 溢价数字来自数据计算，如"az1 DC成本高于 az2/az3 均值 X%"
- 若某 region az1 成本反而低于其他 AZ，如实呈现，不要强行套"az1更贵"的结论
- 若各 AZ 差异极小（<3%），说明该 region AZ 间成本均质，无需迁移

**图表**：堆叠柱状图（3个region），柱顶标注各 AZ 非算力占比（%），配 DC 溢价明细表

### 分析2：高台数-低毛利资源池诊断

**目的**：台数多但毛利低于中位 = 经营异常，需找根因。

**分析步骤：**
1. 以台数中位、毛利中位为分界做四象限，聚焦"高台数-低毛利"象限
2. 对该象限资源池，计算单台成本、单台流水与同资源类型中位的偏差
3. 根因判断：
   - 若单台成本偏高（超中位 X%）→ 成本问题，追溯是否为高成本 AZ
   - 若单台流水偏低 → 流水问题，追溯分配率、CPU 使用率是否偏低
4. 结论中给出实际偏差数字，说明每个异常池的主因

**结论撰写原则：**
- 偏差阈值建议参考：成本超中位 >10% 可认为偏高，分配率低于中位 >15% 可认为偏低
- 若所有高台数池毛利都不差（没有异常），如实说明整体经营健康
- 优先关注主力售卖期，兼顾存量经营期

**图表**：台数×毛利四象限散点图，标注毛利最差的若干异常池

### 分析3：高分配率 + 低 CPU（超卖空间识别）

**目的**：找出分配率高但 CPU 使用率低的资源池，评估超卖可行性。

**分析步骤：**
1. 按阈值筛选（建议：allocation_rate ≥ 0.75 且 cpu_usage ≤ 0.35，可根据数据分布调整）
2. 计算超卖空间 = 分配率 - CPU 使用率，作为超卖潜力指标
3. 优先关注台数多的资源池（超卖收益大、试点风险低）
4. 结合当前毛利率，判断超卖的必要性（毛利已高的池超卖意义相对较小）
5. 基于实际数据给出可落地的超卖建议

**结论撰写原则：**
- 超卖比例建议从实际 CPU 水位推算，而非固定输出"1.2-1.5x"
- 若该资源池 CPU 使用率极低（如 <15%），可考虑更高超卖比，同时提示监控风险
- 若没有明显超卖机会，如实说明

**图表**：分配率 vs CPU 使用率散点图，高亮超卖机会区

## 文件结构

```
scripts/
  generate_report.py        # 编排脚本：加载数据 → 调用分析模块 → 填充模板
  analysis1_az_cost.py      # 分析1：AZ成本差异 + 非算力拆解，run(ms) → dict
  analysis2_problem_pools.py # 分析2：高台数-低毛利诊断，run(ms) → dict
  analysis3_oversell.py     # 分析3：超卖空间识别，run(ms) → dict
  rebuild_data.py           # 重建测试数据
templates/
  report_template.html      # HTML 模板，占位符格式：__KEY__
```

每个分析脚本的 `run(ms)` 接收主力售卖期 DataFrame，返回 dict（含 chart base64、表格 HTML、动态文字）。`generate_report.py` 调用三个模块后用 `str.replace` 填充模板。

## 修改报告的工作流程

1. **理解修改意图** — 确认要改的是图表、分析逻辑、阈值还是样式
2. **定位文件**：
   - 改图表或分析逻辑 → 对应的 `analysis1/2/3_*.py`
   - 改布局/样式 → `templates/report_template.html`
   - 改数据加载或模块调用 → `generate_report.py`
3. **重新运行并打开** — 运行 `generate_report.py`，`open /tmp/cloud_cost_report.html`
4. **向用户确认** — 说明改了哪里，询问是否满意

**重要**：修改报告时，insight 文字中的数字必须从计算变量中插入（f-string），不得硬编码具体数值。若某项分析无异常，结论应说"整体正常，无需重点干预"，而非强行给出建议。
